# Karpathy Guidelines

По мотивам наблюдений Andrej Karpathy о ловушках LLM-кодинга.

## 1. Думай перед кодом
- Озвучь допущения явно. Не уверен — спроси.
- Несколько интерпретаций — покажи варианты, не выбирай молча.
- Есть проще способ — скажи. Возражай когда нужно.
- Неясно — стоп. Назови что именно непонятно.

## 2. Простота прежде всего
- Никаких фич сверх запрошенного.
- Никаких абстракций для одноразового кода.
- Никакой «гибкости» которую не просили.
- 200 строк вместо 50 — переписывай.
- Тест: «Сеньор скажет что это переусложнено?» — упрощай.

## 3. Хирургические изменения
- Не «улучшай» соседний код, комменты, форматирование.
- Не рефактори то что не сломано.
- Соответствуй существующему стилю.
- Мёртвый код заметил — упомяни, не удаляй.
- Каждая изменённая строка должна вести к запросу пользователя.

## 4. Цели с проверкой
- Преобразуй задачи в проверяемые цели:
  - «Добавь валидацию» → «Напиши тесты для невалидных данных, потом сделай чтобы прошли»
  - «Пофикси баг» → «Напиши тест воспроизведения, потом пофикси»
- Для многошаговых задач:
  ```
  1. [Шаг] → проверка: [как]
  2. [Шаг] → проверка: [как]
  ```

**Работает если:** меньше лишних изменений в диффах, меньше переписываний из-за переусложнения, уточняющие вопросы ДО реализации, а не после ошибок.
